## Issue 1 — No global exception handler (B-H1)

### What and why
`grep -r "ControllerAdvice\|ExceptionHandler" backend/src/main/java` returns zero hits. When any controller or service throws an uncaught exception, Spring's default 500 handler serializes the full stack trace and returns it to the client. This leaks internal package structure, library versions, and sometimes query details.

Current ad-hoc handling: controllers throw `ResponseStatusException`, which Spring formats as a minimal JSON body — but only for that specific exception type. Any `IllegalArgumentException`, `JPA` constraint exception, or unhandled `RuntimeException` bypasses this.

### Decision / approach
Add a single `@RestControllerAdvice` class in a new `error` package. It will:
1. Map known exception types to HTTP status + problem-detail body (RFC 7807 shape: `type`, `title`, `status`, `detail`).
2. Log the full stack trace server-side at `ERROR` level for 5xx, `WARN` for 4xx client errors.
3. Return only a safe, generic message to the client — never the raw exception message for 5xx.

RFC 7807 `application/problem+json` is the right content type because it is structured enough for clients to parse and standard enough for future tooling.

### Affected files
| Action | Path |
|---|---|
| Create | `backend/src/main/java/com/meditation/backend/error/GlobalExceptionHandler.java` |
| Create | `backend/src/main/java/com/meditation/backend/error/ProblemDetail.java` (record for the response body) |
| Optionally update | Any controller that currently throws `ResponseStatusException` with a custom message can be cleaned up to throw a domain exception that the handler maps |

### Implementation steps
1. Create package `com.meditation.backend.error`.
2. Create `ProblemDetail` Java record with fields: `String type`, `String title`, `int status`, `String detail`. The `type` can be a URI-like string (`about:blank` is acceptable as a default).
3. Create `GlobalExceptionHandler` annotated `@RestControllerAdvice`:
   - `@ExceptionHandler(ResponseStatusException.class)` → pass through the status code, use `getReason()` as the `detail` field, log at `WARN`.
   - `@ExceptionHandler(MethodArgumentNotValidException.class)` → 400, collect all `FieldError` messages into `detail`, log at `WARN`.
   - `@ExceptionHandler(HttpMessageNotReadableException.class)` → 400 with generic "Malformed request body", log at `WARN`.
   - `@ExceptionHandler(NoHandlerFoundException.class)` → 404, log at `DEBUG`.
   - `@ExceptionHandler(Exception.class)` → 500 with static message "An internal error occurred", log full stack at `ERROR`. Never include exception message or cause in response.
4. Annotate each handler method with `@ResponseStatus` matching its status code and `produces = "application/problem+json"`.
5. Remove or simplify any ad-hoc `try/catch` blocks in controllers that were only there to prevent stack-trace leakage.

### Tests / verification
- Add `GlobalExceptionHandlerTest` using `@WebMvcTest` with a stub controller that deliberately throws each exception type.
- Assert: response `Content-Type` is `application/problem+json`, status matches, body contains `status` and `detail` fields, body does NOT contain the word "Exception" or a stack-trace class name.
- Manual smoke: hit any non-existent endpoint and confirm 404 problem detail, not a Spring Whitelabel error page.

## Issue 2 — Stale-write detection trusts unverified client clock (B-H4, W-H7)

### What and why
`SyncRequestSupport.isStaleMutation(existingUpdatedAt, syncQueuedAtRaw)` compares the server-stored `updated_at` against the client-supplied `X-Meditation-Sync-Queued-At` header, with no check on whether the client's clock is plausible. A client whose clock is 10 minutes behind will always pass the staleness check — every write looks "fresh" because `syncQueuedAt` will be less than `existingUpdatedAt` from the server's perspective, and the guard still fires (`existingUpdatedAt.isAfter(syncQueuedAt)` = true). But a client 10 minutes *ahead* will never trigger staleness detection because all its writes appear to be in the future.

The secondary issue (W-H7): `useTimerSyncEffects` marks queue entries `in-flight` in localStorage *before* the request resolves. A mid-flush reload leaves those entries stranded in `in-flight` forever. The `normalizeHydratedQueue` in `syncQueue.ts` already demotes `in-flight → pending` on load (line 75–86), but there is no staleness threshold: a legitimately in-flight entry on another tab gets reset immediately.

### Decision / approach
**Backend:** Add a configurable clock-skew tolerance (default 300 seconds). In `SyncRequestSupport.isStaleMutation`, before comparing `existingUpdatedAt` to `syncQueuedAt`, validate `|syncQueuedAt − Instant.now()| ≤ skewToleranceSeconds`. If the client clock is too far off, return HTTP 400 with a `problem+json` body indicating the reason. This is not a stale-write; it's a protocol violation.

**Frontend (`syncQueue.ts`):** Add a `stalledAfterMs` field (default 30 000 ms = 30 seconds) to the normalization logic. On queue load, demote `in-flight` entries whose `lastAttemptAt` is older than `stalledAfterMs` back to `pending`. This prevents stranded in-flight entries from blocking the queue forever.

The two fixes are independent; ship them in either order.

### Affected files
| Action | Path |
|---|---|
| Update | `backend/src/main/java/com/meditation/backend/sync/SyncRequestSupport.java` |
| Create | `backend/src/main/resources/application.yml` — add `meditation.backend.sync.clock-skew-tolerance-seconds: 300` |
| Update | `backend/src/main/java/com/meditation/backend/config/` — add a `SyncProperties` config class to bind that key |
| Update | All controllers that call `SyncRequestSupport.isStaleMutation(...)` to pass the tolerance | 
| Update | `src/utils/syncQueue.ts` — `normalizeHydratedQueue` function |
| Update | `src/utils/syncQueue.ts` — export a `STALLED_INFLIGHT_THRESHOLD_MS` constant |

### Implementation steps
**Backend:**
1. Create `SyncProperties` (`@ConfigurationProperties(prefix = "meditation.backend.sync")`) with a `long clockSkewToleranceSeconds` field, defaulting to 300.
2. Change `SyncRequestSupport` to accept a tolerance parameter (or inject `SyncProperties`). If `SyncRequestSupport` remains a static utility, add an overload: `isStaleMutation(Instant existingUpdatedAt, String syncQueuedAtRaw, Duration skewTolerance)`. The no-arg form can call the overloaded form with `Duration.ZERO` for backward compatibility until all callers are updated.
3. In the validation path: after parsing `syncQueuedAt`, compute `|syncQueuedAt.getEpochSecond() - Instant.now().getEpochSecond()| > skewToleranceSeconds`. If true, throw `ResponseStatusException(HttpStatus.BAD_REQUEST, "Client clock is too far from server time; drift detected.")`.
4. Update all callers of `isStaleMutation` in the sync-aware controllers (`CustomPlayController`, `PlaylistController`, `SankalpaController`, `SessionLogController`) to pass the configured tolerance.
5. Add a `Clock` parameter to `SyncRequestSupport` test helpers so the skew check can be tested without wall-clock coupling.

**Frontend:**
1. In `normalizeHydratedQueue` (currently lines 75–86 of `syncQueue.ts`), change the `in-flight → pending` demotion to only trigger if `lastAttemptAt` is older than `STALLED_INFLIGHT_THRESHOLD_MS` (export as a named constant, default 30 000). Entries with no `lastAttemptAt` or a very recent one stay `in-flight` to allow legitimate concurrent syncs to complete.
2. Add a JSDoc comment explaining the rationale (multi-tab reload scenario).

### Tests / verification
**Backend:**
- Unit test in `SyncRequestSupportTest`: client time 400s ahead → no error; client time 400s behind → no error; client time 700s ahead → 400 thrown.
- Controller integration test: PUT with header `X-Meditation-Sync-Queued-At` set to 10 minutes in the future → 400.

**Frontend:**
- Add a test in `syncQueue.test.ts` for `normalizeHydratedQueue`: an `in-flight` entry with `lastAttemptAt` 60 seconds ago → stays `in-flight`; an entry 60 seconds ago when threshold is 30 000 → stays `in-flight` (no clock mock needed); an entry 60 000 ms ago → demoted to `pending`.

## Issue 3 — V14 dropped referential integrity with no replacement (B-H2, B-H3)

### What and why
`V14__drop_session_log_library_foreign_keys.sql` unconditionally drops `fk_session_log_custom_play` and `fk_session_log_playlist`. The commit was made after V10 denormalized `customPlayName` and `playlistName` columns into `session_log`. The likely intent was: "we no longer need the FK because we copy the name". But no service-layer check was added. So:
- `session_log.custom_play_id` and `session_log.playlist_id` can now reference IDs that don't exist.
- There is no invariant anywhere that these IDs are valid.
- Orphaned IDs will silently accumulate as users delete custom plays or playlists.

The denormalized name columns (`customPlayName`, `playlistName`) compound this: if a user renames a custom play, all historical logs keep the old name indefinitely. There is no documented decision about whether this is intentional point-in-time snapshotting or an oversight.

### Decision / approach
Two alternatives:

**Option A (recommended): Restore FKs with `ON DELETE SET NULL` via Flyway migration V16.**
This reinstates database-level referential integrity. When a custom play or playlist is deleted, the corresponding `session_log` fields are set to NULL — the row is not deleted, and the denormalized name columns remain for historical display. This is semantically correct: "the practice happened; it referenced a library item that no longer exists."

**Option B: Service-layer validation.**
In `SessionLogService.saveSessionLog()`, add existence checks for `customPlayId` and `playlistId` before persisting. This is weaker because it only validates at write time, not on delete of the referenced entity. Orphaned IDs created before the fix or via direct DB writes remain undetected.

Option A is chosen because it enforces the invariant durably and handles cascading deletes atomically.

The denormalized names are documented as intentionally point-in-time: add a comment header to V10 migration and to `SessionLogEntity` clarifying this contract.

### Affected files
| Action | Path |
|---|---|
| Create | `backend/src/main/resources/db/migration/V16__restore_session_log_library_foreign_keys.sql` |
| Update | `backend/src/main/java/com/meditation/backend/sessionlog/SessionLogEntity.java` — add comment on `customPlayId` and `playlistId` fields |
| Update | `backend/src/main/resources/db/migration/V10__add_custom_play_fields_to_session_log.sql` — add a header comment (note: this migration is already applied; the comment is for documentation only and does not change the checksum if done as a separate V16 companion) |

### Implementation steps
1. Create `V16__restore_session_log_library_foreign_keys.sql`:
   ```sql
   -- Restores referential integrity dropped in V14.
   -- ON DELETE SET NULL: historical session logs are preserved when a library item is deleted;
   -- customPlayName/playlistName columns retain the point-in-time name snapshot.
   ALTER TABLE session_log
       ADD CONSTRAINT fk_session_log_custom_play
           FOREIGN KEY (custom_play_id) REFERENCES custom_play(id) ON DELETE SET NULL;

   ALTER TABLE session_log
       ADD CONSTRAINT fk_session_log_playlist
           FOREIGN KEY (playlist_id) REFERENCES playlist(id) ON DELETE SET NULL;
   ```
2. Confirm `CustomPlayService` and `PlaylistService` perform hard deletes (not soft deletes). If they do, the `ON DELETE SET NULL` cascade will fire correctly.
3. Add a comment in `SessionLogEntity.java` above `customPlayId` and `playlistId` fields: *"Point-in-time snapshot; nulled by FK cascade when the referenced library item is deleted."*
4. Confirm `SessionLogService.saveSessionLog()` does not need additional validation — the FK will enforce existence at the DB level.

### Tests / verification
- Add `SessionLogForeignKeyTest` (Spring Boot integration test with H2):
  - Create a custom play, log a session referencing it, then delete the custom play.
  - Assert: `session_log.custom_play_id` is NULL; `session_log.custom_play_name` retains the original name.
- Add equivalent test for playlist.
- Run existing `CustomPlayControllerTest` to confirm no regressions.

## Issue 4 — Stacked security risks: Actuator + H2 + wildcard CORS + no auth (B-H7, B-H9, B-H10, X-H1, X-H4)

### What and why
No single item here is necessarily a critical vulnerability in isolation (this is a single-user local deployment). The concern is defense-in-depth: all four defenses are missing simultaneously. If the Mac mini is ever reachable from a non-loopback address (cafe Wi-Fi, misconfigured nginx, VPN, future multi-user scenario), all four gaps apply at once.

The four items and their risk:
- **Wildcard `allowedHeaders("*")`** (WebConfig.java:28): allows an attacker to send arbitrary custom headers cross-origin, which is meaningful combined with mutation endpoints.
- **Actuator exposure**: Spring Boot's default does not restrict which endpoints are exposed. `/actuator/env` exposes environment variables (potential secrets); `/actuator/heapdump` exposes heap contents.
- **H2 with blank password in prod profile**: if deployed with the wrong profile, the H2 database file and console (if re-enabled) are unprotected.
- **No `server.address` restriction**: backend may listen on all interfaces.

### Decision / approach
Fix all four independently — each is a small, self-contained change. They are grouped in this document because they share a "security posture" theme.

No authentication model is proposed here — that is acknowledged as a separate, larger project (X-H1). These fixes harden the existing no-auth model.

### Affected files
| Sub-task | Action | Path |
|---|---|---|
| 4a: Restrict CORS headers | Update | `backend/src/main/java/com/meditation/backend/config/WebConfig.java` |
| 4b: Restrict actuator | Update | `backend/src/main/resources/application.yml` |
| 4c: H2 startup guard | Update | `backend/src/main/java/com/meditation/backend/MeditationBackendApplication.java` |
| 4d: Loopback bind | Update | `backend/src/main/resources/application.yml` |
| 4e: Create prod profile | Create | `backend/src/main/resources/application-prod.yml` |

### Implementation steps

**4a — Restrict CORS headers:**
In `WebConfig.addCorsMappings()`, replace `.allowedHeaders("*")` with an explicit allowlist:
```java
.allowedHeaders(
    "Content-Type",
    "X-Meditation-Sync-Queued-At",
    "X-Requested-With"
)
```
This is safe because these are the only headers the frontend sends. Add a comment in `WebConfig.java` saying: "Keep this list in sync with headers used by the web client; do not restore wildcard."

**4b — Restrict actuator:**
Add to `application.yml` under the `management` key:
```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info
  endpoint:
    health:
      show-details: never
```
This exposes only `/actuator/health` and `/actuator/info`. If richer diagnostics are needed in development, add them in `application-dev.yml` only.

**4c — H2 startup guard:**
In `MeditationBackendApplication`, add a `@Bean` `ApplicationRunner` (not `@PostConstruct` — Spring profiles are active at `ApplicationRunner` time, which is cleaner):
```java
@Bean
ApplicationRunner h2ProdGuard(
        @Value("${spring.datasource.url}") String datasourceUrl,
        Environment env) {
    return args -> {
        if (Arrays.asList(env.getActiveProfiles()).contains("prod")
                && datasourceUrl.contains(":h2:")) {
            throw new IllegalStateException(
                "H2 datasource detected on 'prod' profile. "
                + "Set spring.datasource.url to a production-grade datasource.");
        }
    };
}
```
This does not break local development (which uses no profile or `dev` profile).

**4d — Loopback bind:**
Add to `application.yml`:
```yaml
server:
  address: 127.0.0.1
```
Verify this does not break the existing nginx reverse-proxy setup (nginx should be forwarding to `localhost`). The `mac-mini-production-runbook.md` should be updated to document this constraint.

**4e — Prod profile:**
Create `application-prod.yml` with placeholder overrides. This is the forcing function so future prod deployments have an explicit profile to reach for:
```yaml
# Production profile — override all development defaults here.
# MUST set: spring.datasource.url (non-H2), spring.datasource.username, spring.datasource.password
spring:
  datasource:
    url: ${MEDITATION_DB_URL}
    username: ${MEDITATION_DB_USER}
    password: ${MEDITATION_DB_PASSWORD}
    driver-class-name: org.postgresql.Driver
  jpa:
    hibernate:
      ddl-auto: validate
management:
  endpoints:
    web:
      exposure:
        include: health
```

### Tests / verification
- 4a: Add a `WebConfigTest` that constructs a `MockMvc` CORS preflight for each configured allowed header and one disallowed custom header. Assert the disallowed one is not echoed in the response.
- 4b: Start the context and hit `/actuator/env` — assert 404 (endpoint not mapped).
- 4c: Write a unit test that wires a `prod` active profile + `h2:` URL and asserts `IllegalStateException` on context refresh.
- 4d: Manual verification: after deploy, `curl http://[machine-external-ip]:8080/api/health` should time out or refuse; `curl http://127.0.0.1:8080/api/health` should succeed.

## Issue 5 — Wall-clock timers on web and iOS (W-H6, I-H1)

### What and why
Both the web and iOS clients compute session elapsed time by subtracting wall-clock timestamps. Wall-clock time is not monotonic:
- NTP corrections can move the clock backward mid-session, making elapsed time go negative.
- On iOS, `Timer.publish(every: 1, on: .main)` is scheduled on the main run loop, which can stall under heavy UI load, silently skipping ticks.
- After device sleep or background suspension, the wall-clock jumps forward but accumulated time was not tracked — catch-up logic in both clients partially compensates but is not complete.

**Web consequence:** `getActiveSessionElapsedSeconds` (time.ts:14) computes `Math.floor((nowMs - session.lastResumedAtMs) / 1000)`. If NTP adjusts `Date.now()` backward between two ticks, elapsed time decreases — the timer visually goes backward. `Math.max(0, ...)` guards against a visible negative, but silently stalls the timer.

**iOS consequence:** Missed ticks on a wall-clock `Timer.publish` under main-thread pressure means the displayed timer lags, and the interval bell fires at the wrong moment.

### Decision / approach

**Web:**
Store a parallel `lastResumedAtPerformanceMs: number | null` in `ActiveSession` alongside the existing `lastResumedAtMs`. `performance.now()` is guaranteed monotonic within a browsing context. Use it exclusively for elapsed-time calculation; use `Date.now()` only for log timestamps and display-mode labels.

The `performance.now()` epoch resets on page reload, which is fine — `lastResumedAtPerformanceMs` is only meaningful within a single page session. On page reload / session restore from storage, the catch-up logic that already exists (foreground catch-up in `foregroundCatchUp.ts`) continues to handle the inter-session gap.

**iOS:**
Replace `Timer.publish(every: 1, on: .main)` with a `DispatchSourceTimer` scheduled on a background serial queue. The timer fires on a background queue and dispatches the UI update to the main actor via `Task { @MainActor in ... }`. Use `ContinuousClock` (monotonic, available from iOS 16) for elapsed accounting; keep `Date()` for display strings and log records.

|---|---|
| Update | `src/types/timer.ts` — add `lastResumedAtPerformanceMs: number \| null` to `ActiveSession` |
| Update | `src/features/timer/time.ts` — change `getActiveSessionElapsedSeconds` and `getActiveSessionElapsedMilliseconds` to use `performanceMs` parameter |
| Update | `src/features/timer/timerReducer.ts` — when recording `lastResumedAtMs` on RESUME, also record `lastResumedAtPerformanceMs: performance.now()` |
| Update | `src/features/timer/TimerContext.tsx` — pass `performance.now()` to elapsed calculations |
| Verify | `src/features/timer/foregroundCatchUp.ts` — existing catch-up logic uses `Date.now()`; document why wall-clock is acceptable there (catch-up is for cross-session gaps, not within-session precision) |

**iOS:**

### Implementation steps

2. In `timerReducer.ts`, find all places that set `lastResumedAtMs` to `Date.now()` (i.e., on RESUME action). Alongside, set `lastResumedAtPerformanceMs: performance.now()`.
3. Update `getActiveSessionElapsedSeconds` and `getActiveSessionElapsedMilliseconds` in `time.ts` to accept a second parameter `nowPerformanceMs: number` and use it against `session.lastResumedAtPerformanceMs`. The `nowMs` parameter is retained for non-elapsed uses.
4. Update all callers in `TimerContext.tsx` and elsewhere to pass `performance.now()` as the elapsed parameter.
5. For tests, `performance.now()` can be mocked via `vi.stubGlobal('performance', { now: () => ... })`.

**iOS:**

2. Add a `private var clockTimer: DispatchSourceTimer?` property.
3. In the equivalent of `startClock()`, create:

   ```swift
   timer.schedule(deadline: .now(), repeating: .milliseconds(200), leeway: .milliseconds(50))
   timer.setEventHandler { [weak self] in

           self?.now = Date()

       }
   }
   timer.resume()
   self.clockTimer = timer
   ```

5. Keep `Date()` for display labels and for the `endedAt` field written to `AppSnapshot`.
6. Cancel `clockTimer` in `deinit` and in `stopSession()`.

## Issue 6 — Three monster files (W-H1, W-H2, I-H5)

### What and why
These three files concentrate the majority of behavioral complexity in the app. Every new feature gets appended to one of them, making them harder to reason about and impossible to test in isolation. This is a multi-sprint refactor; the design must be agreed on before starting to avoid expensive merge conflicts.

| File | Lines | Concern |
|---|---|---|
| `src/features/timer/TimerContext.tsx` | 1,372 | Timer, custom play, playlist, session log, hydration, audio wiring, visibility handlers |
| `src/features/timer/useTimerSyncEffects.ts` | 822 | Hydration + reconciliation for 4 entity types, ~18 dedup refs |
| `ios-native/MeditationNative/App/ShellViewModel.swift` | 1,678 | Persistence, sync, timer SM, custom play SM, playlist SM, notifications, audio delegation |

### Decision / approach

**TimerContext.tsx — split into domain-sliced contexts:**

The file currently builds one monolithic `TimerContextValue` with ~50 methods. The fix is to split into multiple context providers, each with a narrow concern. Consumers import only what they need; a change to custom play logic does not force re-renders of timer-only consumers.

Proposed split:
- `TimerStateContext` — exposes the reducer `state` and `dispatch`. Read by components that need raw state.
- `TimerActionsContext` — exposes the stable dispatch-helper functions (start, pause, stop, etc.). These are `useCallback`-memoized against `dispatch`, so they change only when dispatch changes.
- `CustomPlayContext` — custom play collection state (load, save, delete) and active-run state.
- `PlaylistRuntimeContext` — playlist collection state and active-playlist-run state.
- `SessionLogContext` — session log list, save, and manual log operations.
- `TimerSettingsContext` — settings load/save only.

The existing `TimerContext.tsx` becomes a thin `TimerProvider` that composes these five providers. The existing `timerContextObject.ts` is split into `timerStateContext.ts`, `customPlayContext.ts`, `playlistRuntimeContext.ts`, `sessionLogContext.ts`, `timerSettingsContext.ts`.

**useTimerSyncEffects.ts — hydration factory:**

The file has four near-identical blocks that hydrate `customPlays`, `playlists`, `sessionLogs`, and `timerSettings` from API + storage. Extract a `createCollectionHydrator<T>` factory function that takes:
```typescript
interface CollectionHydratorOptions<T> {
  storageLoader: () => T[];
  apiLoader: () => Promise<T[]>;
  reducer: { actionType: string; toAction: (items: T[]) => TimerAction };
  queueReconciler?: (storedItems: T[], apiItems: T[]) => SyncQueueEntry[];
  dedupeRef: React.MutableRefObject<Set<string>>;
}
```
Each of the four collections becomes one `createCollectionHydrator(...)` call. The file shrinks from ~822 lines to ~200.

**ShellViewModel.swift — extract three session coordinators:**

Three state machines currently live inside the ViewModel: timer, custom play, and playlist. Each has its own start/pause/stop/complete lifecycle, audio side effects, and notification scheduling. Extract:

- `TimerSessionCoordinator` — owns `ActiveTimerSession?`, drives the clock, triggers audio/notification at completion, coordinates background keepalive.
- `CustomPlaySessionCoordinator` — owns `ActiveCustomPlaySession?`, drives audio player, handles completion.
- `PlaylistSessionCoordinator` — owns `ActivePlaylistSession?`, drives per-item playback, advances items.

`ShellViewModel` becomes a thin `@Observable`/`@MainActor` object that:
- Holds references to the three coordinators.
- Owns `@Published` properties that mirror coordinator state for SwiftUI.
- Delegates all domain logic to the coordinator.
- Owns sync and persistence concerns (which are already somewhat separate via `AppSyncService` / `JSONFileStore`).

### Implementation steps

This is a large refactor. Recommended execution order to minimize risk:

## Issue 7 — No frontend ErrorBoundary (W-H8)

### What and why
`grep` for `ErrorBoundary|componentDidCatch|getDerivedStateFromError` in `src/` returns zero hits. React renders the component tree synchronously during reconciliation. If any component throws during render — including components deep inside the provider tree — React will unmount the entire application and display a blank screen. Without an `ErrorBoundary`, there is no recovery path and no error is surfaced to the user.

### Decision / approach
Add a class-component `ErrorBoundary` (class components are required for `getDerivedStateFromError`/`componentDidCatch` in React 19). Wrap `<App />` in `src/main.tsx` with the boundary. Optionally also wrap individual routes for finer-grained recovery (recommended but can be done incrementally).

The recovery UI should show a brief message and a "Reload" button. It should also call a configurable `onError` callback so that a future error-reporting sink can be added without changing the boundary code.

### Affected files
| Action | Path |
|---|---|
| Create | `src/app/ErrorBoundary.tsx` |
| Update | `src/main.tsx` — wrap `<App />` |
| Optionally update | `src/App.tsx` — wrap each `<Route>` element |

### Implementation steps
1. Create `src/app/ErrorBoundary.tsx`:
   ```tsx
   interface Props {
     children: ReactNode;
     onError?: (error: Error, info: ErrorInfo) => void;
     fallback?: ReactNode;
   }

   interface State { hasError: boolean; error: Error | null }

   export class ErrorBoundary extends Component<Props, State> {
     state: State = { hasError: false, error: null };

     static getDerivedStateFromError(error: Error): State {
       return { hasError: true, error };
     }

     componentDidCatch(error: Error, info: ErrorInfo) {
       this.props.onError?.(error, info);
       console.error('[ErrorBoundary]', error, info.componentStack);
     }

     render() {
       if (this.state.hasError) {
         return this.props.fallback ?? (
           <div role="alert">
             <p>Something went wrong.</p>
             <button onClick={() => window.location.reload()}>Reload</button>
           </div>
         );
       }
       return this.props.children;
     }
   }
   ```
2. In `src/main.tsx`, wrap `<React.StrictMode><App /></React.StrictMode>` with `<ErrorBoundary>`.
3. Apply basic styling to the fallback UI so it doesn't look completely broken; it just needs to be functional.
4. Optionally: in `src/App.tsx`, wrap the router outlet or each major route in a per-route boundary with a lighter fallback ("This section failed to load; other parts of the app still work").

### Tests / verification
- Add `ErrorBoundary.test.tsx`: render a child that throws during render, assert the fallback UI appears with a reload button.
- Test that `onError` callback is called with the thrown error.
- Manual test: throw in a leaf component (temp `throw new Error('test')`) inside a route and confirm the app does not go blank.

## Issue 9 — Sync queue dedup silently loses deletes (W-M7)

### What and why
`enqueueSyncQueueEntry` in `src/utils/syncQueue.ts` (lines 127–146) deduplicates by `(entityType, recordId)` regardless of the operation type. The filter removes all existing entries for a `(entityType, recordId)` pair before adding the new entry.

The bug scenario:
1. User deletes record R. `DELETE(R)` is queued.
2. A background reconciliation or re-render causes `UPSERT(R)` to be enqueued (e.g., hydration re-queues the sync entry for an in-memory-modified record).
3. The dedup removes the `DELETE(R)` from the queue. Only `UPSERT(R)` remains.
4. The backend receives only the UPSERT and never the DELETE — the record persists when it should have been deleted.

This is distinct from the legitimate "delete then re-create" flow. In that flow, the user explicitly creates a new record with the same ID — in this app, record IDs are client-generated UUIDs, so a re-created record would have a new ID.

### Decision / approach
Change the dedup logic to be operation-aware:
- **Incoming DELETE**: removes all existing entries for the key (both `upsert` and `delete`) and adds the new delete. A delete always supersedes everything.
- **Incoming UPSERT**: removes existing `upsert` entries for the key, but **preserves** any existing `delete` entries. The rationale: if a delete is pending, it was intentional and a background upsert should not override it.

The one edge case is a legitimate "re-create after delete": user deletes R, then immediately creates a new record and the client assigns it the same ID. This should not happen in practice (client IDs are UUIDs generated fresh), but if it does, the behavior is: the DELETE stays in the queue ahead of the UPSERT, the backend receives DELETE then UPSERT, and the record ends up re-created — which is correct.

### Affected files
| Action | Path |
|---|---|
| Update | `src/utils/syncQueue.ts` — `enqueueSyncQueueEntry` function |

### Implementation steps
1. In `enqueueSyncQueueEntry` (line 127), replace the single `dedupedQueue` filter with operation-aware logic:
   ```typescript
   let dedupedQueue: SyncQueueEntry[];
   if (nextEntry.operation === 'delete') {
     // A delete supersedes any pending upserts or prior deletes for this record.
     dedupedQueue = queue.filter(
       (entry) => buildSyncQueueRecordKey(entry.entityType, entry.recordId) !== nextRecordKey
     );
   } else {
     // An upsert deduplicates against other upserts but preserves a pending delete.
     dedupedQueue = queue.filter(
       (entry) =>
         buildSyncQueueRecordKey(entry.entityType, entry.recordId) !== nextRecordKey ||
         entry.operation === 'delete'
     );
   }
   ```
2. Add a JSDoc comment on `enqueueSyncQueueEntry` explaining the dedup semantics.
3. Update the return statement to use `dedupedQueue`.

### Tests / verification
Add the following cases to `syncQueue.test.ts`:
- **Delete after upsert**: queue has `[UPSERT(R)]`, enqueue `DELETE(R)` → queue is `[DELETE(R)]`.
- **Upsert after delete**: queue has `[DELETE(R)]`, enqueue `UPSERT(R)` → queue is `[DELETE(R), UPSERT(R)]` (delete preserved, upsert added after).
- **Upsert after upsert**: queue has `[UPSERT(R)]`, enqueue `UPSERT(R)` → queue is `[UPSERT(R)]` (deduplicated, only one remains).
- **Delete after delete**: queue has `[DELETE(R)]`, enqueue `DELETE(R)` → queue is `[DELETE(R)]` (one delete, deduplicated).
- **Mixed keys**: enqueueing for record S does not affect entries for record R.

## Issue 10 — iOS bell reliability unproven (I-H1 through I-H4, I-H6)

### What and why
`EXECPLAN-ios-native-bell-reliability.md` exists because this is a known live defect. The end bell (the sound that signals a meditation session has ended) is unreliable in the background. The root causes are multiple and interact:

1. **I-H1 Wall-clock timer**: `Timer.publish(every: 1, on: .main)` is skipped when the main thread is busy or the app is backgrounded. Addressed by Issue 5 above; resolved by `DispatchSourceTimer`.
2. **I-H3 Conditional keepalive**: `SilentBackgroundAudioKeepAlive` is activated only for a subset of bell-emitting conditions. A custom play *with* a recording does not trigger keepalive, so the audio session deactivates when the recording ends, and the end bell fires into a deaf session.
3. **I-H4 No interruption handling**: Phone calls, Siri, other audio interruptions drop the audio session without recovery. The bell is silently lost.
4. **I-H6 No `deactivate(.notifyOthersOnDeactivation)` in teardown**: When the audio session ends abnormally, other apps' ducked audio is not restored.
5. **I-H2 Strong `self` captures in Tasks**: Retain cycles mean `ShellViewModel` outlives its intended lifetime; Tasks fire on a deallocated object context.

### Decision / approach
Address each root cause independently, roughly in this order (each is testable in isolation):

1. **Fix strong captures first** (I-H2) — these are dangerous and affect all async operations, not just audio.
2. **Fix keepalive coverage** (I-H3) — make keepalive fire whenever any bell-emitting condition could apply.
3. **Add interruption handling** (I-H4) — register for `AVAudioSession.interruptionNotification` in each player.
4. **Fix teardown** (I-H6) — call `deactivate(.notifyOthersOnDeactivation)` in all audio session teardown paths.
5. **The wall-clock timer** (I-H1) is already addressed in Issue 5 above.

### Affected files

|---|---|---|
| I-H2 | Update | `ios-native/MeditationNative/App/ShellViewModel.swift` — add `[weak self]` to Task captures at lines ~369, 599, 909, 1249, 1265–1267, 1555, 1573 |

   ```swift
       .sink { [weak self] notification in
           guard let type = notification.userInfo?[AVAudioSessionInterruptionTypeKey] as? UInt,
                 let interruptionType = AVAudioSession.InterruptionType(rawValue: type) else { return }
   ```
